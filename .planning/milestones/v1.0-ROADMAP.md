# Roadmap: Christ Fensterhandel Konfigurator

## Overview

Vom leeren Next.js-Projekt zum produktiven Fenster-Konfigurator in 7 Phasen: Zuerst das Fundament (Payload CMS + alle Collections), dann der 10-Step-Konfigurator als Herzstück, danach Warenkorb + Preisberechnung + Anfrage-Absenden als kompletter Kauffluss, dann Admin- und Kunden-Dashboards mit Rollen, dann externe Integrationen (Stripe + N8N/E-Mail), dann Website-Features (Page Builder, Mehrsprachigkeit, DSGVO), und zuletzt Docker-Deployment auf Coolify.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

## Phase Details

### Phase 1: Fundament
**Goal**: Das komplette Datenmodell steht — alle CMS Collections sind angelegt, miteinander verknuepft, konditional filterbar und mit realistischen Testdaten befuellt
**Depends on**: Nothing (first phase)
**Requirements**: SETUP-01, SETUP-02, SETUP-03, CMS-01, CMS-02, CMS-03, CMS-04, CMS-05, CMS-06, CMS-07, CMS-08, CMS-09, CMS-10, CMS-11, CMS-12, CMS-13, CMS-14, CMS-15, CMS-16, CMS-17
**Success Criteria** (what must be TRUE):
  1. Next.js App startet lokal mit Payload Admin Panel und verbindet sich mit PostgreSQL
  2. Alle 15+ Collections sind im Admin sichtbar, Felder editierbar, Beziehungen navigierbar
  3. Konditionale Filterung funktioniert: Material-Auswahl schraenkt verfuegbare Profile ein, Produkttyp schraenkt Fluegelanzahl ein
  4. Seed-Script fuellt alle Collections mit realistischen Drutex-Daten (Profile, Farben, Preise) auf einen Befehl
  5. Anfragen-Collection hat Status-Feld mit Workflow-Werten und Status-Historie als unveraenderliches Log
**Plans**: 3 plans

Plans:
- [x] 01-01: Projekt-Setup (Next.js + Payload CMS + PostgreSQL + Tailwind + Shadcn)
- [x] 01-02: Produkt-Collections (produkttypen, materialien, profile, fluegelanzahl, zusatzlichter, oeffnungsarten, fensterformen)
- [x] 01-03: Ausstattungs-Collections (farben, dichtungsfarben, verglasungen, schallschutz, sicherheitsglas, glasdekore, sprossen, extras)
- [x] 01-04: Business-Collections (preisregeln, rabattcodes, anfragen, status_historie) + Konditionale Filterung + Seed-Script

### Phase 2: Konfigurator-Pipeline
**Goal**: Ein Kunde kann einen kompletten 10-Step Fenster-Konfigurator durchlaufen — jede Auswahl filtert die naechsten Optionen intelligent
**Depends on**: Phase 1
**Requirements**: KONF-01, KONF-02, KONF-03, KONF-04, KONF-05, KONF-06, KONF-07, KONF-08, KONF-09, KONF-10, KONF-11, KONF-12, KONF-13, KONF-14, KONF-15
**Success Criteria** (what must be TRUE):
  1. Landing Page zeigt 3 Konfigurator-Karten, Klick auf Fenster/Balkontuer oeffnet den Konfigurator unter /konfigurator/fenster
  2. Alle 10 Steps sind durchlaufbar: Produkttyp, Material, Profil, Fluegel, Oeffnungsart, Form, Masse, Farben, Verglasung/Extras, Zusammenfassung
  3. Konditionale Ketten-Logik funktioniert: Wahl von Material filtert Profile, Wahl von Produkttyp filtert Fluegelanzahl, Profil bestimmt Min/Max Masse
  4. Zurueck-Navigation setzt abhaengige Felder zurueck (Material aendern -> Profil reset), Fortschrittsanzeige zeigt aktuellen Step
  5. Zusammenfassung (Step 10) zeigt alle gewaehlten Optionen korrekt mit Preisvorschau
**Plans**: 5 plans

Plans:

- [x] 02-02-PLAN.md — UI-Shell (Landing Page, Routing, 3-Spalten Layout, Sidebar, Navigation, Preview, UI-Components)
- [x] 02-03-PLAN.md — Steps 1-5 (Produkttyp, Material, Profil, Fluegel, Oeffnungsart) mit konditionaler Filterung + SVG Fluegel
- [x] 02-04-PLAN.md — Steps 6-9 (Form, Masse, Farben, Verglasung/Extras) mit RHF/Zod + SVG Farbe/Sprossen

### Phase 3: Kauffluss
**Goal**: Ein Kunde kann konfigurierte Produkte in den Warenkorb legen, Preise sehen, Rabattcodes anwenden und eine vollstaendige Anfrage absenden
**Depends on**: Phase 2
**Requirements**: CART-01, CART-02, CART-03, CART-04, CART-05, CART-06, PREIS-01, PREIS-02, PREIS-03, PREIS-04, PREIS-05, SEND-01, SEND-02, SEND-03, SEND-04
**Success Criteria** (what must be TRUE):
  1. Konfiguriertes Produkt landet im Warenkorb, Stueckzahl ist aenderbar, Produkt kann bearbeitet oder geloescht werden
  2. Warenkorb zeigt korrekte Zwischensumme; ein zweiter Konfigurator-Durchlauf fuegt weiteres Produkt hinzu
  3. Server-seitige Preisberechnung liefert verbindlichen Preis (nicht manipulierbar), Rabattcode wird validiert (gueltig/abgelaufen/aufgebraucht/min-bestellwert)
  4. Kontaktformular mit allen Pflichtfeldern + Datenschutz-Checkbox, Server-seitige Zod-Validierung
  5. Anfrage wird in CMS gespeichert mit Status "NEU" und eingefrorener Konfigurations-Snapshot, Kunde sieht Danke-Seite
**Plans**: 3 plans

Plans:
- [ ] 03-01-PLAN.md — Warenkorb-Store + Cart Page UI + Navigation Badge (CART-01 bis CART-06)
- [ ] 03-02-PLAN.md — Server-Preisberechnung, Rabattcode-Validierung, Zod-Schemas, Anfrage-Nummer (PREIS-01 bis PREIS-05)
- [ ] 03-03-PLAN.md — Anfrage-Flow (Discount-UI, Kontaktformular, Zusammenfassung, Submit-API, Danke-Seite) (SEND-01 bis SEND-04)

### Phase 4: Dashboards und Rollen
**Goal**: Admin verwaltet Anfragen mit Status-Workflow und Historie, Kunden sehen eigene Anfragen — mit rollenbasiertem Zugriff
**Depends on**: Phase 3
**Requirements**: SEC-01, SEC-02, SEC-03, ADMIN-01, ADMIN-02, ADMIN-03, ADMIN-04, ADMIN-05, ADMIN-06, KUND-01, KUND-02, KUND-03, KUND-04
**Success Criteria** (what must be TRUE):
  1. Admin sieht alle Anfragen mit Filter (Status, Datum, Kategorie), Suche (Name, E-Mail, Nr.) und kann Detailansicht oeffnen
  2. Status-Workflow funktioniert: NEU -> IN BEARBEITUNG -> BESTAETIGT -> BEZAHLT -> ABGESCHLOSSEN + RUECKFRAGE/ABGELEHNT, jede Aenderung wird in der Historie protokolliert
  3. Interne Notizen sind pro Anfrage erfassbar und nur fuer Admin/Mitarbeiter sichtbar
  4. Kunde kann sich registrieren/einloggen und sieht unter /kunden/dashboard nur eigene Anfragen mit Status-Timeline
  5. Access Control greift: Admin darf alles, Mitarbeiter bearbeitet Anfragen, Viewer nur lesen, Kunde sieht nur eigenes
**Plans**: 3 plans

Plans:
- [x] 04-01: Security-Grundlagen (Zod auf allen Endpoints, Rollen-Feld, Access Control Policies)
- [x] 04-02: Admin-Dashboard (Anfragen-Liste, Filter, Suche, Detailansicht, Status-Workflow, Historie, Notizen)
- [x] 04-03: Kunden-Auth und Dashboard (Login/Register, geschuetzte Route, eigene Anfragen, Status-Timeline)

### Phase 5: Externe Integrationen
**Goal**: Bezahlung ueber Stripe und automatisierte E-Mails ueber N8N funktionieren Ende-zu-Ende
**Depends on**: Phase 4
**Requirements**: PAY-01, PAY-02, PAY-03, N8N-01, N8N-02, N8N-03, N8N-04, N8N-05
**Success Criteria** (what must be TRUE):
  1. Aus einer bestaetigten Anfrage kann eine Stripe Checkout Session erstellt werden, Testzahlung mit 4242-Karte funktioniert
  2. Stripe Webhook setzt Anfrage-Status automatisch auf "BEZAHLT"
  3. Neue Anfrage triggert Webhook an N8N, N8N sendet formatierte E-Mail an Firma und Bestaetigungs-E-Mail an Kunde
  4. Status-Aenderung triggert passende E-Mail an Kunden (z.B. "Ihre Anfrage wird bearbeitet")
  5. Webhook-Endpoint ist mit Secret gesichert (ohne Secret -> 401)
**Plans**: 2 plans

Plans:
- [ ] 05-01-PLAN.md — Stripe Integration (Checkout Session, Bezahl-Button, Webhook, Status auf BEZAHLT)
- [ ] 05-02-PLAN.md — N8N Integration (Webhook-Utility, afterChange Hooks, Docker Setup, Fehler-Badge)
