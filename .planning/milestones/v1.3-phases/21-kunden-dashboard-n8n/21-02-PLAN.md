```markdown
# N8N Workflow-Dokumentation: Kunden-E-Mail-Trigger

## Uebersicht

Das Christ Fensterhandel System sendet bei **jeder** Status-Aenderung einen Webhook an N8N. N8N entscheidet anhand des `customer_facing` Flags, ob eine E-Mail an den Kunden gesendet wird.

## Webhook-Payload

Bei jedem Status-Wechsel sendet das System folgenden Payload an die konfigurierte N8N_WEBHOOK_URL:

| Feld | Typ | Beschreibung |
|------|-----|--------------|
| event_type | string | "neue_anfrage" / "status_aenderung" / "zahlung_eingegangen" |
| anfrage_id | string | Payload CMS Anfrage ID |
| anfrage_nummer | string | Menschenlesbare Anfrage-Nummer (z.B. "ANF-2026-001") |
| status.neu | string | Neuer Status-Key |
| status.alt | string | Vorheriger Status-Key |
| kunde.name | string | Kundenname |
| kunde.email | string | Kunden-E-Mail-Adresse |
| gesamtbetrag | number | Gesamtpreis in EUR |
| produkt_anzahl | number | Anzahl konfigurierter Produkte |
| customer_facing | boolean | true wenn Kunde benachrichtigt werden soll |
| kunden_text | string | Kundenfreundlicher Status-Text (z.B. "Ihre Anfrage wird bearbeitet.") |
| kunden_phase | string/null | Aktuelle Kunden-Phase ("Anfrage"/"Angebot"/"Zahlung"/"Produktion"/"Lieferung") oder null bei Endstatus |
| stripe_checkout_url | string/null | Nur bei zahlung_eingegangen Events |

## Empfohlener N8N-Workflow

### Schritt 1: Webhook-Trigger
- Typ: Webhook
- Methode: POST
- URL: `{N8N_BASE_URL}/webhook/christ-fensterhandel`
- Authentifizierung: x-webhook-secret Header pruefen gegen gespeichertes Secret

### Schritt 2: Customer-Facing Filter
- Typ: IF-Node
- Bedingung: `{{ $json.customer_facing === true }}`
- True-Pfad: weiter zu E-Mail
- False-Pfad: Ende (interne Status-Aenderung, keine E-Mail)

### Schritt 3: E-Mail-Template Auswahl
- Typ: Switch-Node
- Feld: `{{ $json.status.neu }}`
- Cases (14 kundenrelevante Statuse):
  - `neu` -> Eingangsbestaetigungs-E-Mail
  - `rueckfrage` -> Rueckfrage-E-Mail mit Hinweis auf Timeline
  - `angebot_versendet` -> Angebots-E-Mail
  - `bestaetigt` -> Angebots-Bestaetigungs-E-Mail
  - `zahlungslink_versendet` -> Zahlungslink-E-Mail (stripe_checkout_url verwenden)
  - `bezahlt` -> Zahlungsbestaetigungs-E-Mail
  - `in_produktion` -> Produktionsstart-E-Mail
  - `hersteller_problem` -> Hersteller-Problem-E-Mail
  - `versandbereit` -> Versandbenachrichtigung
  - `geliefert` -> Lieferbestaetigungs-E-Mail
  - `abgeschlossen` -> Abschluss-E-Mail
  - `storniert` -> Stornierungs-E-Mail
  - `zahlungsproblem` -> Zahlungsproblem-E-Mail
  - `reklamation` -> Reklamations-Bestaetigungs-E-Mail

### Schritt 4: E-Mail senden
- Typ: Send Email (SMTP oder E-Mail-Service)
- An: `{{ $json.kunde.email }}`
- Betreff: Abhaengig vom Template (z.B. "Ihre Anfrage ANF-2026-001 -- Statusaktualisierung")
- Body: `{{ $json.kunden_text }}` als Haupttext, ergaenzt um Anrede und Signatur
- Kunden-Phase im Footer: "Aktueller Schritt: {{ $json.kunden_phase }}"

## Umgebungsvariablen

| Variable | Beschreibung |
```
