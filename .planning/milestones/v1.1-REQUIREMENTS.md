## v1.1 Requirements -- Admin-Panel Umbau: Profile Hub + UX

**Core Value:** Admin kann Profile als zentralen Hub fuer alle Produkt- und Ausstattungs-Zuordnungen pflegen -- mit Undo/Redo Sicherheitsnetz und lueckenloser Aenderungshistorie.

### Profile Hub

- [x] **HUB-01**: Admin kann in profile.ts 13 hasMany-Relationship-Felder (erlaubte_*) in zwei Tabs (Kombinationen / Ausstattung) pflegen
- [x] **HUB-02**: Alle 13 Relationship-Felder haben filterOptions ({ aktiv: { equals: true } }) und admin.allowCreate: true
- [x] **HUB-03**: Alle 13 neuen Felder haben maxDepth: 0 um Response-Explosion zu verhindern
- [x] **HUB-04**: Bestehendes Einzel-Feld `material` (single relation) bleibt unveraendert
- [x] **HUB-05**: Admin sieht Warnung/Badge wenn Hub-Felder eines Profils noch leer sind (Incomplete Profile)

### Migration

- [x] **MIG-01**: Migration-Script befuellt erlaubte_farben automatisch aus farben.erlaubte_materialien + profile.material Abgleich
- [x] **MIG-02**: Migration ist idempotent (ueberschreibt nur leere Felder, laesst bereits gesetzte unberuehrt)
- [x] **MIG-03**: Migration laeuft paginiert (PAGE_SIZE 100) und loggt pro Profil was aktualisiert wurde

### Filter-Logik

- [x] **FILT-01**: filters.ts nutzt Hub-Felder (erlaubte_*) als primaere und einzige Quelle fuer Konfigurator-Optionen (Steps 4-6, 8-9). Ketten-Filter auf Steps 1-3, 7 bleiben unveraendert (vor Profil-Auswahl).
- [x] **FILT-02**: Hub-Feld leer = Kategorie wird NICHT angezeigt (kein Legacy-Fallback). Admin MUSS alle Hub-Felder befuellen bevor USE_HUB=true. Spec: `docs/todos/009_2026-03-18_hub-ersetzt-ketten-mapping.md`
- [x] **FILT-03**: Feature-Flag USE_HUB in filters.ts schaltet zwischen altem Ketten-Code und neuem Hub-Code (kein Mischbetrieb). Nach Validierung: alten Code + Flag komplett loeschen.
- [x] **FILT-04**: Konfigurator zeigt keine Regressions -- alle 10 Steps funktionieren mit USE_HUB=true (befuellte Profile) und USE_HUB=false (alter Code) fehlerfrei
- [x] **FILT-05**: store.ts Profile-Fetch Depth wird von depth=2 auf depth=1 reduziert
- [x] **FILT-06**: Validation Script (`npm run validate:hub-fields`) prueft alle Profile auf vollstaendige Hub-Feld-Befuellung vor USE_HUB=true Switch

### Undo/Redo

- [x] **UNDO-01**: Proof-of-Concept: form.getFields() + replaceState() funktioniert mit Relationship-Feldern (vor voller Implementierung verifizieren)
- [x] **UNDO-02**: UndoRedoProvider als globaler Admin-Provider registriert (admin.components.providers)
- [x] **UNDO-03**: ProfileEditToolbar mit Undo/Redo Buttons im Profile Edit-View (beforeDocumentControls)
- [x] **UNDO-04**: Keyboard-Shortcuts Cmd+Z (Undo) und Cmd+Shift+Z (Redo) funktionieren im Profile Edit-View
- [x] **UNDO-05**: Undo/Redo Stack ist Session-scoped und per Dokument isoliert (collectionSlug:id Key)
- [x] **UNDO-06**: Snapshots werden debounced (300ms) bei Formular-Aenderungen erstellt

### Edit-History

- [x] **HIST-01**: Neue edit_history Collection mit Feldern: collection, doc_id, event, diff (JSON), editor (User-Relation), timestamp
- [x] **HIST-02**: beforeChange Hook setzt last_edited_by auf aktuellen User (kein afterChange + update = kein Infinite Loop)
- [x] **HIST-03**: afterChange Hook erstellt edit_history Eintrag mit Diff (geaenderte Feldnamen) und Editor
- [x] **HIST-04**: afterChange Hook nutzt req.context.skipEditHistory Guard um Infinite Loops zu verhindern
- [x] **HIST-05**: History-Panel im Profile Edit-View zeigt letzte 50 Aenderungen mit Timestamp, Event und Editor
- [x] **HIST-06**: last_edited_by Sidebar-Feld (readOnly) zeigt letzten Bearbeiter im Profile Edit-View

### Tech-Debt

- [x] **DEBT-01**: filters.ts Step 9 filtert verglasungen, schallschutz, sicherheitsglas, glasdekore, sprossen, extras ueber Hub-Felder (statt ungefiltert)
- [x] **DEBT-02**: filters.ts filtert ueberall nach aktiv === true (aktuell kein aktiv-Check)
- [x] **DEBT-03**: Bei generate:types pruefen ob Fensterforman/Sicherheitsgla Typos korrigiert sind, ggf. Collection Labels anpassen
- [x] **DEBT-04**: dichtungsfarben Collection wird ueber Hub-Feld erlaubte_dichtungsfarben auf Profile gefiltert (aktuell nie gefiltert)
- [x] **DEBT-05**: filterOptions { aktiv: { equals: true } } auf allen Hub-Relationship-Feldern konsistent setzen (inkl. Collections ohne bisherige aktiv-Nutzung)
- [x] **DEBT-06**: Versions-Config Entscheidung dokumentieren: KEIN versions:drafts in v1.1 (Research-Empfehlung wegen _status Migration-Risiko)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Webflow-basierte Loesung | Komplett auf Next.js + Payload migriert |
| Supabase / externe DB | PostgreSQL direkt, kein externer Service |
| Retool/Softr Dashboard | Payload Admin Panel ist das Dashboard |
| Echtzeit-Chat | Nicht Teil des Konfigurators, hohe Komplexitaet |
| Mobile App | Web-first, responsive reicht fuer v1 |
| Garagentore/Raffstore | Eigene Konfiguratoren, spaeter |
| Payload Versions/Drafts auf Profile | _status Migration-Risiko: bestehende Profile werden unsichtbar (null != published). ADR: docs/entscheidungen/001. Deferred to v1.2 (VRSN-01 mit Pre-Migration). |
| last_published_by Tracking | Nur sinnvoll mit Drafts/Publish Workflow, v1.2 |
| Frontend-UI Aenderungen am Konfigurator | Nur filters.ts Backend-Logik, keine neuen Steps oder Layouts |
| Legacy-Fallback / Dual-Pfad-Logik | Hub ersetzt Ketten komplett, kein Parallelbetrieb (docs/todos/009) |
| Legacy-Relationen entfernen | Erst nach Hub-Validierung, nicht vorher anfassen |
| Before/After Value Diffs in History | Feldnamen-Tracking reicht fuer v1.1 |
| Undo/Redo fuer andere Collections | Profile-only in v1.1, Architektur erlaubt spaetere Extension |
