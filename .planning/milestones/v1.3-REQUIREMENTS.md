# Requirements: Christ Fensterhandel v1.3

**Defined:** 2026-03-23
**Core Value:** Kunden konfigurieren Fenster/Tueren Schritt fuer Schritt mit intelligenter Hub-Filterung -- Christ bekommt strukturierte Anfragen ins Dashboard

## v1.3 Requirements

Requirements fuer Milestone v1.3: Bestellungsflow + Admin UX Redesign.

### Status-System

- [x] **STAT-01**: status-config.ts als Single Source of Truth fuer STATUS_COLORS (hex), STATUS_LABELS, STATUS_TAILWIND (Klassen), STATUS_CUSTOMER_TEXT
- [x] **STAT-02**: Alle 4 bestehenden Komponenten importieren Farben/Labels aus status-config.ts (keine lokale Duplikation)
- [x] **STAT-03**: 15+ neue Status-Werte im Anfragen-Collection Select-Feld mit PostgreSQL-Migration
- [x] **STAT-04**: Erweiterte Status-Transitions mit linearem Hauptflow und Abzweigungen (Rueckfrage, Stornierung, Hersteller-Problem)
- [x] **STAT-05**: Admin-Status zu Kunden-Text Mapping mit 5-Phasen-Modell (Anfrage/Angebot/Zahlung/Produktion/Lieferung)
- [x] **STAT-06**: last_status_change_at Feld auf Anfragen-Collection mit automatischem Update via beforeChange Hook

### Collection-Felder

- [x] **FELD-01**: Hersteller-Felder auf Anfragen: hersteller_bestellnummer (Text), lieferdatum_erwartet (Date), hersteller_notizen (Textarea), hersteller_antwort (Textarea)
- [x] **FELD-02**: Stornierung-Felder auf Anfragen: stornierung_grund (Textarea), rueckerstattung_betrag (Number), rueckerstattung_status (Select: ausstehend/durchgefuehrt/abgelehnt)
- [x] **FELD-03**: customer_facing Flag auf WebhookPayload Interface damit N8N nur bei kundenrelevanten Status-Wechseln E-Mails triggert

### Admin UX

- [x] **ADMN-01**: admin-custom.css fuer strukturelle Admin-Styles (ersetzt Inline-Style-Ceiling bei 486+ Zeilen)
- [x] **ADMN-02**: Attention Bar Component (volle Breite, Anfrage-Nr, Status-Badge, Wartezeit mit Farb-Codierung, Gesamtpreis, Produkt-Zusammenfassung)
- [x] **ADMN-03**: Aktions-Leiste mit Splitbutton-Pattern (primaere Status-Aktion + Dropdown fuer Abzweigungen)
- [x] **ADMN-04**: 2-Spalten Layout mit Produkte (60%) und Tab-Panel (40%) fuer Kontakt/Timeline/Notizen
- [x] **ADMN-05**: Stueckzahl-Badge pro Produktkarte (gross, fett, farbig) mit Einzelpreis x Menge = Gesamtpreis Darstellung
- [x] **ADMN-06**: Anfrage-Detail-View komplett umgebaut mit Attention Bar + Aktions-Leiste + 2-Spalten-Tabs
- [x] **ADMN-07**: Anfragen-Liste mit Filter-Tabs (Alle/Offen/Rueckfrage/In Produktion/Abgeschlossen)
- [x] **ADMN-08**: Wartezeit-Spalte in Anfragen-Liste mit Farb-Codierung (gruen/gelb/orange/rot)
- [x] **ADMN-09**: Attention-Score Sortierung (Wartezeit x Status-Gewicht, dringendste oben)
- [x] **ADMN-10**: Kontextabhaengige Quick-Actions pro Status (primaere Aktion + sekundaere Actions basierend auf Quick-Actions-Tabelle aus Todo 017)

### N8N Integration

- [x] **N8N-01**: E-Mail-Trigger bei 10 kundenrelevanten Status-Aenderungen als N8N Webhook Events (Eingang, Rueckfrage, Angebot, Zahlungslink, Zahlung eingegangen, In Produktion, Versandbereit, Geliefert, Storniert, Hersteller-Problem) -- customer_facing Flag (FELD-03) steuert Ausloesung

### Kunden-Dashboard

- [x] **KUND-01**: Kunden sehen vereinfachte Status-Texte statt interner Admin-Statuse
- [x] **KUND-02**: 5-Phasen Fortschrittsbalken (Anfrage -> Angebot -> Zahlung -> Produktion -> Lieferung) mit aktuellem Schritt markiert

## v2 Requirements

Deferred zu zukuenftigem Release.

### Reklamation
- **REKL-01**: Reklamation-Notizen Textarea auf Anfragen (V1-Loesung)
- **REKL-02**: Eigene Reklamation-Collection mit Fotos, Status, Loesung

### Kunden Self-Service
- **SELF-01**: Kunde kann Stornierung selbst einreichen
- **SELF-02**: Kunde kann Reklamation mit Fotos einreichen

### Automatisierung
- **AUTO-01**: Stripe automatische Rueckerstattung bei Stornierung
- **AUTO-02**: E-Mail-Templates als eigene Collection (statt N8N-hardcoded)

## Out of Scope

| Feature | Reason |
|---------|--------|
| hersteller_bestaetigt_mit_vorbehalt Status | Research-Empfehlung: zu komplex fuer v1.3, entscheiden bei Phase-2-Start |
| Kanban-Board fuer Statuse | Anti-Feature bei 15+ Statuse -- Filter-Tabs sind besser |
| Separate payment_status + fulfillment_status Felder | Korrekt aber hohe Migration-Kosten, sequentieller Flow reicht fuer v1.3 |
| zahlungsproblem Status (Stripe Rueckbuchung) | Setzt Stripe-Rueckbuchungs-Webhook voraus, erst mit AUTO-01 |
| Reklamation-Notizen | User hat Reklamation fuer v2 deferred |
