## Phase Details

### Phase 17: Status-Config Centralization
**Goal**: Alle Status-Metadaten (Farben, Labels, Gruppen, Kunden-Texte) leben in einer einzigen Datei -- keine lokale Duplikation mehr
**Depends on**: Phase 16 (v1.2 complete)
**Requirements**: STAT-01, STAT-02
**Success Criteria** (what must be TRUE):
  1. `src/lib/status-config.ts` existiert mit Exports fuer STATUS_COLORS (hex), STATUS_LABELS, STATUS_TAILWIND (Klassen), STATUS_CUSTOMER_TEXT und STATUS_CUSTOMER_PHASE
  2. Alle 6 bestehenden Komponenten (status-workflow, status-timeline, anfrage-detail-view, dashboard-overview, kunden/status-timeline, kunden/gast-tracking-form) importieren Farben und Labels ausschliesslich aus status-config.ts
  3. `grep -r "STATUS_COLORS\|STATUS_LABELS" src/` liefert null lokale Definitionen ausserhalb status-config.ts
  4. Bestehendes Verhalten (Status-Badge-Farben, Timeline-Labels) ist identisch zu vorher -- keine visuelle Regression
**Plans**: 2 plans

Plans:
- [ ] 17-01-PLAN.md -- Create status-config.ts with all exports and unit tests (STAT-01)
- [ ] 17-02-PLAN.md -- Migrate all 6 consumer files to import from status-config.ts (STAT-02)

### Phase 18: Statuses, Transitions und Collection-Felder
**Goal**: Das Anfragen-Datenmodell bildet den vollstaendigen Bestellungsflow ab -- von Eingang bis Lieferung, inklusive Abzweigungen fuer Rueckfrage, Stornierung und Hersteller-Probleme
**Depends on**: Phase 17
**Requirements**: STAT-03, STAT-04, STAT-06, FELD-01, FELD-02, FELD-03
**Success Criteria** (what must be TRUE):
  1. Anfragen-Collection hat 15+ Status-Werte im Select-Feld und die PostgreSQL-Datenbank akzeptiert alle neuen Enum-Werte ohne Fehler
  2. Status-Transitions erlauben nur gueltige Uebergaenge (z.B. `bezahlt` kann zu `an_hersteller` wechseln, aber nicht zurueck zu `neu`) -- ungueltige Transitions werden server-seitig abgelehnt
  3. Hersteller-Felder (Bestellnummer, Lieferdatum, Notizen, Antwort) sind sichtbar sobald Status `an_hersteller` oder spaeter erreicht ist; Stornierung-Felder (Grund, Rueckerstattungsbetrag, Rueckerstattungsstatus) sind sichtbar bei Status `storniert`
  4. `last_status_change_at` wird automatisch bei jedem Status-Wechsel aktualisiert und zeigt den Zeitpunkt des letzten Uebergangs
  5. WebhookPayload enthaelt `customer_facing: boolean` Flag basierend auf einer EMAIL_TRIGGER_STATUSES Liste in status-config.ts
**Plans**: 3 plans

Plans:
- [ ] 18-01-PLAN.md -- Extend status-config.ts to 20 statuses with all flat maps and updated tests (STAT-03)
- [ ] 18-02-PLAN.md -- Extend status-transitions.ts and n8n-webhook.ts with 20-status transitions and enriched WebhookPayload (STAT-04, FELD-03)
- [ ] 18-03-PLAN.md -- Extend anfragen.ts with 20 select options, new fields, hook modifications (STAT-03, STAT-06, FELD-01, FELD-02)

### Phase 19: Admin Detail View Redesign
**Goal**: Admin-Mitarbeiter sehen auf einen Blick den Zustand einer Anfrage und koennen die naechste Aktion mit einem Klick ausfuehren -- statt durch eine unstrukturierte Seite zu scrollen
**Depends on**: Phase 18
**Requirements**: ADMN-01, ADMN-02, ADMN-03, ADMN-04, ADMN-05, ADMN-06, ADMN-10
**Success Criteria** (what must be TRUE):
  1. `admin-custom.css` ist via Payload Config eingebunden und strukturelle Admin-Styles (Layout, Grid, Spacing) sind als CSS-Klassen definiert statt als Inline-Styles
  2. Attention Bar zeigt am oberen Rand: Anfrage-Nummer, farbigen Status-Badge, Wartezeit mit Farb-Codierung (gruen/gelb/orange/rot), Gesamtpreis und Produkt-Zusammenfassung
  3. Splitbutton zeigt die primaere Status-Aktion als grossen Button und sekundaere Aktionen (Abzweigungen wie Rueckfrage, Stornierung) im Chevron-Dropdown -- Aktionen sind kontextabhaengig pro aktuellem Status
  4. Detail-View hat 2-Spalten-Layout: Produktkarten mit Stueckzahl-Badge (60%) links, Tab-Panel mit Kontakt/Timeline/Notizen (40%) rechts
  5. Stueckzahl-Badge auf jeder Produktkarte zeigt Menge gross und farbig (32x32px) mit Einzelpreis x Menge = Gesamtpreis Darstellung
**Plans**: 4 plans

Plans:
- [x] 19-01-PLAN.md -- Data layer + CSS foundation: QUICK_ACTIONS map, formatCurrency extraction, helper functions, admin-custom.css classes (ADMN-01, ADMN-10)
- [x] 19-02-PLAN.md -- Sub-components: AttentionBar, Splitbutton, ProductCard, TabPanel (ADMN-02, ADMN-03, ADMN-04, ADMN-05)
- [ ] 19-03-PLAN.md -- Integration: Rewrite anfrage-detail-view.tsx composing all sub-components + visual checkpoint (ADMN-06)
- [ ] 19-04-PLAN.md -- Gap closure: Extract inline styles to CSS classes, fix admin-content wrapper (ADMN-01)

### Phase 20: Admin List View Redesign
**Goal**: Admin-Mitarbeiter sehen in der Anfragen-Liste sofort welche Anfragen Aufmerksamkeit brauchen -- sortiert nach Dringlichkeit, filterbar nach Workflow-Phase
**Depends on**: Phase 19
**Requirements**: ADMN-07, ADMN-08, ADMN-09
**Success Criteria** (what must be TRUE):
  1. Filter-Tabs (Alle/Offen/Rueckfrage/In Produktion/Abgeschlossen) gruppieren Anfragen nach semantischen Status-Gruppen -- Tab-Wechsel filtert die Liste sofort
  2. Wartezeit-Spalte zeigt fuer jede Anfrage die Zeit seit dem letzten Status-Wechsel mit Farb-Codierung (gruen < 24h, gelb 1-3 Tage, orange 3-7 Tage, rot > 7 Tage)
  3. Standard-Sortierung ist Attention-Score (Wartezeit x Status-Gewicht) -- dringendste Anfragen stehen oben, ohne dass der Admin manuell sortieren muss
**Plans**: 2 plans

Plans:
- [x] 20-01-PLAN.md -- Data layer + CSS foundation: STATUS_WEIGHT, LIST_TAB_FILTERS, list-view-helpers, BEM CSS classes (ADMN-07, ADMN-08, ADMN-09)
- [x] 20-02-PLAN.md -- UI components: AnfragenListView, ListMenu, collection config registration + visual checkpoint (ADMN-07, ADMN-08, ADMN-09)

### Phase 21: Kunden-Dashboard + N8N
**Goal**: Kunden sehen ihren Bestellstatus in verstaendlicher Sprache mit Fortschrittsbalken und erhalten E-Mails bei relevanten Status-Aenderungen
**Depends on**: Phase 18 (Status-Mapping aus Phase 17, Statuse aus Phase 18)
**Requirements**: STAT-05, KUND-01, KUND-02, N8N-01
**Success Criteria** (what must be TRUE):
  1. Kunden sehen im Dashboard vereinfachte deutsche Status-Texte (z.B. "Ihre Anfrage wird bearbeitet") statt interner Admin-Statuse (z.B. "in_bearbeitung") -- kein interner Status-Key ist irgendwo im Kunden-Portal sichtbar
  2. 5-Phasen-Fortschrittsbalken (Anfrage - Angebot - Zahlung - Produktion - Lieferung) zeigt den aktuellen Schritt farbig markiert und vergangene Schritte als erledigt
  3. Bei 10 kundenrelevanten Status-Aenderungen (Eingang, Rueckfrage, Angebot, Zahlungslink, Zahlung eingegangen, In Produktion, Versandbereit, Geliefert, Storniert, Hersteller-Problem) wird ein N8N Webhook mit `customer_facing: true` ausgeloest
  4. Interne Status-Aenderungen (z.B. `in_bearbeitung` zu `an_hersteller`) loesen keinen Kunden-E-Mail-Trigger aus
**Plans**: 2 plans

Plans:
- [x] 21-01-PLAN.md -- New components: ProgressStepper, StatusBanner, pulse-slow CSS animation + unit tests (KUND-02, STAT-05)
- [x] 21-02-PLAN.md -- Integration: Wire stepper/banner into 3 kunden components, replace internal labels, N8N verification + docs + visual checkpoint (KUND-01, KUND-02, N8N-01, STAT-05)
