## Phase Details

### Phase 7: Data Model Foundation
**Goal**: Profile-Collection wird zum zentralen Hub mit 13 hasMany-Relationship-Feldern in zwei Tabs, und die edit_history Collection existiert fuer Audit-Logging
**Depends on**: Phase 6 (v1.0 complete)
**Requirements**: HUB-01, HUB-02, HUB-03, HUB-04, HIST-01
**Success Criteria** (what must be TRUE):
  1. Admin sieht im Profile Edit-View zwei Tabs "Kombinationen" und "Ausstattung" mit je zugeordneten erlaubte_*-Feldern
  2. Alle 13 Relationship-Felder zeigen nur aktive Eintraege (filterOptions aktiv=true) und erlauben Inline-Erstellung (allowCreate)
  3. Profile-API-Responses enthalten fuer die 13 neuen Felder nur IDs, keine verschachtelten Objekte (maxDepth: 0 verifiziert)
  4. Das bestehende Einzel-Feld "material" funktioniert weiterhin unveraendert
  5. Collection edit_history existiert in der Datenbank mit Feldern collection, doc_id, event, diff, editor, timestamp
**Plans**: 2 plans

**Research notes:**
- maxDepth: 0 auf allen 13 neuen Feldern setzen (verhindert Response-Explosion)
- Nach Schema-Aenderungen: `npm run generate:importmap && npm run generate:types`
- DB-Dump VOR Schema-Migration erstellen (push:true Sicherheit)

Plans:
- [x] 07-01-PLAN.md -- Profile Hub: 13 hasMany-Relationship-Felder in 2 Tabs + Tests
- [x] 07-02-PLAN.md -- edit_history Collection + Config-Registration + Type-Generation

### Phase 9: Filter Logic Refactor
**Goal**: Hub-Felder ersetzen Ketten-Filter KOMPLETT (Steps 4-6, 8-9) -- kein Legacy-Fallback, USE_HUB Feature-Flag fuer Rollback zu altem Code
**Depends on**: Phase 8
**Requirements**: FILT-01, FILT-02, FILT-03, FILT-04, FILT-05, FILT-06, DEBT-01, DEBT-02
**Success Criteria** (what must be TRUE):
  1. Konfigurator-Steps 4-6 und 8-9 laden Optionen AUSSCHLIESSLICH aus den erlaubte_*-Feldern des gewaehlten Profils (Hub ersetzt Ketten-Filter)
  2. Hub-Feld leer = Kategorie wird NICHT angezeigt (kein Legacy-Fallback). Steps 1-3 und 7 bleiben unveraendert.
  3. Feature-Flag USE_HUB in filters.ts schaltet zwischen altem Ketten-Code und neuem Hub-Code (kein Mischbetrieb)
  4. Validation Script prueft alle Profile auf vollstaendige Hub-Feld-Befuellung (npm run validate:hub-fields)
  5. store.ts Profile-Fetch nutzt depth=1 statt depth=2 (keine Response-Explosion)
  6. Alle 10 Konfigurator-Steps funktionieren fehlerfrei mit USE_HUB=true (befuellte Profile) und USE_HUB=false (alter Code)
**Plans**: 2 plans

**Canonical refs:**
- `docs/todos/009_2026-03-18_hub-ersetzt-ketten-mapping.md` — Vollstaendiges Step-fuer-Step Mapping (Source of Truth)
- `docs/todos/008_2026-03-15_admin-panel-umbau-plan.md` — Abschnitt 3 (Filter-Logik)

**Research notes:**
- HOECHSTES RISIKO Phase -- manuelle QA durch alle 10 Steps Pflicht
- Hub ersetzt Ketten KOMPLETT, kein Parallelbetrieb (docs/todos/009 ist Source of Truth)
- getHubField() Helper: leer = null = Kategorie nicht anzeigen
- Steps 1-3 + 7 behalten Ketten-Logik (vor Profil-Auswahl bzw. bereits am Profil)
- Step 9 (Verglasung etc.) ueber Hub filtern = DEBT-01 erledigt
- aktiv===true Check ueberall einbauen = DEBT-02 erledigt
- Rollout: USE_HUB=false deployen → Admin befuellt Profile → validate:hub-fields → USE_HUB=true
- Nach Validierung: alten Ketten-Code und Feature-Flag komplett loeschen (v1.2 FLAG-01 + LEGC-01)

Plans:
- [x] 09-01-PLAN.md -- getHubField helper + USE_HUB feature flag + filters.ts refactor + store.ts aktiv-filter + STEP_DEPENDENCIES
- [x] 09-02-PLAN.md -- Step component updates + validation script + manual QA checkpoint

### Phase 10: Undo/Redo
**Goal**: Admin hat ein Sicherheitsnetz beim Bearbeiten von Profilen -- jede Aenderung kann per Button oder Keyboard-Shortcut rueckgaengig gemacht und wiederhergestellt werden
**Depends on**: Phase 7
**Requirements**: UNDO-01, UNDO-02, UNDO-03, UNDO-04, UNDO-05, UNDO-06
**Success Criteria** (what must be TRUE):
  1. Admin kann im Profile Edit-View Undo (Cmd+Z) und Redo (Cmd+Shift+Z) nutzen und sieht Formular-Aenderungen zurueckgesetzt/wiederhergestellt
  2. Undo/Redo Buttons in der Toolbar zeigen korrekt enabled/disabled je nach Stack-Zustand
  3. Wechsel zwischen verschiedenen Profil-Dokumenten setzt den Undo/Redo-Stack zurueck (kein Cross-Document Bleed)
  4. Relationship-Felder (erlaubte_*) werden korrekt per Undo/Redo wiederhergestellt (nicht nur einfache Text/Number-Felder)
**Plans**: 2 plans

**Research notes:**

- getFields() verwenden, NICHT getData() (Type-Mismatch mit replaceState)
- Stack per collectionSlug:id keyen (useDocumentInfo)
- Snapshots mit 300ms Debounce
- Falls replaceState fuer Relationships fehlschlaegt: Fallback auf per-field dispatchFields
